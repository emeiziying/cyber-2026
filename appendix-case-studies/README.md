# 附录：真实案例库

> **适合人群：** 工程师
> **学习目标：** 通过完整案例把“会看文档”转成“会做任务”，并形成可复用的复盘材料

---

## 为什么要有案例库？

大多数团队引入 AI 时，真正的断层不在概念理解，而在这里：

- 会看路线图，但不会把它用到真实任务里
- 会复制模板，但不知道什么时候该用哪一个
- 会让 AI 写代码，但不会做完整复盘

案例库的作用，就是把一条任务链路从输入、执行、验证到复盘走完整。

---

## 推荐优先补充的 3 类案例

### 案例 1：修一个真实 bug

目标：练习“问题描述 -> 代码定位 -> 最小修复 -> 回归测试”的闭环。

建议输出：
- 问题描述
- 根因分析
- 修复说明
- 回归测试
- 复盘结论

### 案例 2：为现有模块补测试

目标：练习“读取现有实现 -> 识别边界条件 -> 生成并验证测试”的闭环。

建议输出：
- 被测对象说明
- 测试策略
- 新增测试点
- 漏测风险

### 案例 3：做一次 PR Review

目标：练习“读取上下文 -> 识别高风险点 -> 输出结构化 review finding”的闭环。

建议输出：
- 变更摘要
- 风险点列表
- 建议优先处理项
- 最终评审结论

---

## 建议案例结构

每个案例都建议按以下结构写：

1. 背景与目标
2. 输入材料
3. 执行过程
4. 输出结果
5. 验证方式
6. 复盘与经验沉淀

模板见：
- [`templates/case-study-template.md`](./templates/case-study-template.md)

# 一个最小治理样例：发布前的风险审查

## 任务背景

后端团队收到一个需求：修改登录后 token 刷新逻辑，将原来的固定有效期改为滑动窗口过期策略，并上线到测试环境验证。

这个改动看起来范围不大，但它直接影响认证链路——一旦出错，所有登录用户都会受影响。

---

## 风险判断

| 维度 | 判断 | 说明 |
|------|------|------|
| 影响范围 | 高 | 涉及所有登录用户的 session 管理 |
| 可逆性 | 中 | 可以回滚部署，但已刷新的 token 无法撤回 |
| 数据敏感度 | 高 | 直接操作认证凭证 |
| AI 参与边界 | 受限 | 允许分析和草稿，不允许跳过 review |

**结论：** 不属于低风险，需要走完整审批流程。

---

## AI 允许做什么 / 不允许做什么

**允许：**
- 分析现有 token 刷新代码，梳理调用链路
- 起草修复方案，列出改动文件清单
- 补充回归测试用例
- 生成风险说明文档

**不允许：**
- 直接提交到主分支
- 跳过 code review 推进合并
- 自行决定上线时间窗口
- 修改与认证无关的代码（防止范围蔓延）

---

## 必要确认点（三道关卡）

### 关卡 1：编码前确认

- [ ] 改动范围是否只限 token 刷新逻辑？
- [ ] 是否会影响第三方 OAuth 回调？
- [ ] 滑动窗口的时间参数是否已确认？

### 关卡 2：合并前确认

- [ ] 是否补了回归测试，覆盖以下路径？
  - 正常登录 → 使用 → token 自动续期
  - token 过期 → 重新登录
  - 并发请求时的 token 竞争
- [ ] 是否有 reviewer 确认过认证链路无遗漏？
- [ ] 是否更新了接口文档（如果 token 格式或返回结构有变化）？

### 关卡 3：发布前确认

- [ ] 是否有明确的回滚方案？（回滚步骤、预计耗时、数据影响）
- [ ] 测试环境是否已验证至少 24 小时？
- [ ] 是否通知了前端团队配合验证？

---

## 验收问题清单

| 验证路径 | 预期行为 | 是否通过 |
|----------|----------|----------|
| 正常登录 → 操作 → 等待接近过期 → 继续操作 | token 自动续期，不中断 | |
| 正常登录 → 长时间不操作 → 超过最大窗口 | 要求重新登录 | |
| token 过期后调用接口 | 返回 401，前端跳转登录页 | |
| 退出登录 | token 立即失效，无法复用 | |
| 并发刷新请求 | 只有一个成功，其余拿到相同新 token | |
| 错误态（伪造 token、空 token） | 返回明确错误码，不泄露内部信息 | |

---

## 复盘：这个过程教会我们什么

这个案例的关键不是"AI 能不能写好 token 刷新"，而是：

1. **风险分级决定流程**——同样是改几行代码，改配色和改认证链路需要完全不同的审批力度
2. **AI 的参与边界要提前划清**——不是事后发现问题再限制，而是开始前就明确
3. **确认点要落到检查清单**——口头说"记得测一下"不如一张可勾选的清单可靠

如果团队发现类似的高风险改动反复出现，应该把这套审查模式沉淀到项目级 Rules 或 Skill 中，让后续任务自动带上风险审查步骤。

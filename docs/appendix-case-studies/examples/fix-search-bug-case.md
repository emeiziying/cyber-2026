# 案例：修复搜索结果为空的 bug

> 本案例按照 <a href="/downloads/appendix-case-studies/templates/case-study-template.md" download>case-study-template.md</a> 的结构编写，展示一个从问题发现到经验沉淀的完整闭环。

---

## 1. 背景与目标

### 项目信息

- 项目：内部后台管理系统（Node.js + Fastify + SQLite）
- 模块：笔记搜索功能
- 团队：3 人研发小组，已使用 AI 辅助开发 4 周

### 问题描述

用户反馈：搜索中文关键词时，部分已有笔记搜不到。搜索英文关键词正常。

### 任务目标

定位并修复中文搜索失败的根因，确保修复后不破坏现有 CRUD 和英文搜索行为。

---

## 2. 输入材料

提交给 AI 的任务描述：

> 笔记搜索功能存在 bug：用中文关键词搜索时，部分包含该关键词的笔记搜不到。
>
> 已知信息：
> - 搜索接口：GET /notes/search?q=关键词
> - 数据库：SQLite，使用 LIKE 做模糊匹配
> - 英文搜索正常，中文部分失败
> - 数据库中确认存在包含目标关键词的记录
>
> 约束：
> - 只改搜索相关代码，不动 CRUD 接口
> - 不引入新依赖（不要换全文索引引擎）
> - 修复后补回归测试

---

## 3. 执行过程

### 第一轮：定位问题

AI 的分析路径：

1. 读取 `src/repositories/note-repository.js` 中的搜索实现
2. 发现 SQL 查询使用 `WHERE title LIKE '%' || ? || '%' OR content LIKE '%' || ? || '%'`
3. 进一步检查发现数据库连接没有设置 `PRAGMA encoding`
4. 测试发现：SQLite 默认的 LIKE 对多字节字符（中文）的匹配依赖底层 C 库的 locale 设置

**根因确认：** SQLite 的 LIKE 操作在默认配置下对非 ASCII 字符的大小写和匹配行为不一致。部分中文字符在内部编码比较时被错误跳过。

### 第二轮：确认修复方案

AI 提出两个方案：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A | 用 `INSTR()` 替代 `LIKE` | 简单直接，字节级匹配准确 | 不支持通配符模式 |
| B | 自定义 SQLite 的 `like` 函数 | 完全兼容 LIKE 语法 | 实现复杂，需要 C 扩展 |

**决策：** 选择方案 A。当前场景只需要子串匹配，不需要通配符，方案 A 的简单性更适合。

### 第三轮：实现修复

改动文件清单：

```
src/repositories/note-repository.js  — 将 LIKE 替换为 INSTR()
test/note-service.test.js            — 新增 4 个中文搜索测试用例
```

核心改动：

```javascript
// 修复前
const sql = `SELECT * FROM notes WHERE title LIKE '%' || ? || '%' OR content LIKE '%' || ? || '%'`;

// 修复后
const sql = `SELECT * FROM notes WHERE INSTR(title, ?) > 0 OR INSTR(content, ?) > 0`;
```

### 第四轮：验证

| 测试场景 | 预期 | 结果 |
|----------|------|------|
| 搜索英文关键词 "hello" | 返回匹配笔记 | ✅ 通过 |
| 搜索中文关键词 "笔记" | 返回匹配笔记 | ✅ 通过 |
| 搜索中英混合 "test笔记" | 返回匹配笔记 | ✅ 通过 |
| 搜索不存在的关键词 | 返回空数组 | ✅ 通过 |
| 空关键词 | 返回全部笔记 | ✅ 通过 |
| 现有 CRUD 测试 | 全部通过 | ✅ 无回归 |

---

## 4. 输出结果

### 修复摘要

- **根因：** SQLite 的 LIKE 操作对中文字符匹配不可靠
- **修复方式：** 用 INSTR() 替代 LIKE 做子串匹配
- **影响范围：** 仅搜索查询，不影响 CRUD
- **新增测试：** 4 个用例，覆盖中文、混合、空值场景

### PR 描述

```markdown
## 修复中文搜索失败的问题

### 问题
搜索中文关键词时部分笔记搜不到，原因是 SQLite LIKE 对多字节字符匹配不可靠。

### 修复
将搜索查询从 LIKE 改为 INSTR()，确保字节级子串匹配准确。

### 测试
新增 4 个测试用例，覆盖中文、中英混合、空关键词场景。
已有 CRUD 测试全部通过，无回归。
```

## 6. 复盘与经验沉淀

### AI 做对了什么

- 定位根因的思路正确：先检查 SQL 语句 → 再检查数据库配置 → 最后确认是 LIKE 的多字节问题
- 提供了两个方案并对比优缺点，没有直接上最复杂的方案
- 修复后主动列出了"没改什么"，帮助 reviewer 确认范围

### AI 做偏了什么

- 第一次建议的方案 B（自定义 LIKE 函数）过于复杂，需要人工判断后选择方案 A
- 没有主动考虑空关键词的边界场景，需要人工追问后补充

### 沉淀到 Rules 的内容

在项目 `CLAUDE.md` 中新增一条规则：

```markdown
- 涉及 SQLite 的文本匹配查询时，优先使用 INSTR() 而非 LIKE，以避免多字节字符匹配问题
```

### 沉淀到 Skill 的内容

在 fix-bug Skill 模板中补充一个检查步骤：

```markdown
修复完成后，确认以下边界场景：
- 中文 / 多字节字符输入
- 空输入
- 特殊字符输入（引号、百分号、反斜杠）
```

---

## 总结

这个案例展示了一个典型的"小 bug、大教训"场景：

1. **问题本身不复杂**——改一行 SQL 就修好了
2. **定位过程需要系统思维**——不是"搜索不工作"，而是"特定编码下的匹配行为不一致"
3. **修复后的沉淀比修复本身更有价值**——一条 Rule + 一个 Skill 检查步骤，让同类问题不再重复发生

# 团队工作流与质量控制

> **适合人群：** 工程师 + Reviewer + Tech Lead
> **学习目标：** 把个人的 AI 使用经验沉淀成团队可执行的协作流程、review 标准、测试义务和持续复盘机制，并明确这些资产应由各项目自己维护

---

## 9.1 本章解决什么问题

个人用 AI 时，只要自己觉得快、顺手、结果还行，通常就够了。  
但团队协作里，真正的问题会马上冒出来：

- 任务入口格式不统一
- review 标准因人而异
- 有人补测试，有人完全不补
- Rules、Skills、Hooks 没人维护

本章的重点，是把“个人技巧”升级成“团队流程”。

---

## 9.2 为什么这件事重要

如果团队没有统一工作流，AI 用得越多，协作噪音往往越大：

- 需求理解不一致
- diff 质量参差不齐
- review 成本上升
- 经验沉淀只停留在聊天记录里

所以第九章真正要解决的不是“让每个人都更会用 AI”，而是让团队能在共同约束下稳定协作。

这一章讲的是治理原则和维护方式，不把这里当成统一实现仓。团队真正要维护的，是各项目自己的 Rules、Skills、Hooks、Agent 和检查清单。

---

## 9.3 核心概念与边界

### 一个最小团队工作流

1. 任务入口标准化
2. AI 执行或辅助分析
3. 自检与测试
4. PR Review
5. 合并 / 回滚准备
6. 迭代复盘

### 团队里至少要明确的角色

| 角色 | 主要职责 |
|------|----------|
| 任务发起者 | 把目标、范围、边界和验收说清楚 |
| 执行者 | 使用 AI 推进任务，并补自检与说明 |
| Reviewer | 重点看需求理解、边界、测试和风险说明 |
| 维护者 | 维护各项目自己的 Rules、Skills、Hooks、Agent 资产和模板 |

### 边界：不是把所有动作都流程化

团队流程的目标是降低噪音和风险，不是让每一步都变成表单负担。  
真正值得标准化的，是那些**反复发生、且一旦缺失就会明显影响协作质量**的环节。

### 边界：这里不是统一实现仓

本套方法论在这里提供的是维护原则、清单模板和复盘方法。
真实需要长期维护的 `CLAUDE.md`、项目级 Skills、Hooks、Agent 配置和运行时约束，应由各项目在各自工程里维护。

## 9.4 一个最小场景与反例

### 正例：统一任务入口后再让 AI 执行

一个中等复杂度任务，先写出：

- 背景和目标
- 范围与不包含项
- 关键技术约束
- 验收标准
- 参考文件

然后再让 AI 开始推进。

### 反例：直接把模糊需求扔给 AI

“把搜索功能优化一下，顺便看看有没有别的问题。”

这类输入的问题在于：

- 目标不清楚
- 范围失控
- 无法验收
- review 时也没人知道本来应该做到哪一步

团队里最值得先标准化的，通常就是这种任务入口。

---

## 9.5 实操方法：让团队流程真正可执行

### 1. 标准化任务入口

至少统一这些字段：

- 目标
- 范围
- 技术约束
- 参考资料
- 验收标准

### 2. 明确测试义务

建议提前约定最小规则：

- 修 bug：必须补回归测试
- 新功能：至少覆盖 happy path 和一个失败路径
- 重构：至少跑回归测试，关键模块优先补自动化保护

### 3. 统一 review 关注点

AI 相关 review 不该只看“代码像不像人写的”，而应重点看：

- 是否正确理解需求
- 是否遗漏边界条件
- 是否有解释不清的新增逻辑
- 是否说明了风险和验证方式

### 4. 明确维护责任人

至少要知道谁负责：

- 各项目自己的 `CLAUDE.md`
- 各项目自己的 Skills / Agent 配置
- 各项目自己的 MCP / Hook 配置
- 模板和清单

---

## 9.6 复盘如何反写到团队资产

轻量复盘至少回答下面 4 个问题：

1. 本迭代 AI 最省时间的场景是什么？
2. 最容易出错的场景是什么？
3. 哪类问题应该补进 Rules？
4. 哪类重复流程值得做成 Skill / Hook？

如果复盘只停留在口头总结，团队经验不会进入下一轮迭代。  
真正有价值的是把结论反写到共享方法资产或项目资产里。

---

## 9.7 常见误区 / 风险提醒

| 误区 | 更稳妥的做法 |
|------|--------------|
| “流程越细越好” | 只标准化真正影响协作质量的关键环节 |
| “review 还是按以前那套就行” | AI 时代更要看边界、验证和说明 |
| “测试是执行者个人习惯” | 测试义务应该是团队约定，而不是个人喜好 |
| “Rules / Skills 有人想起了再改” | 没有明确维护责任人，资产会很快失效 |

**风险提醒：**  
团队工作流最容易失败的原因，不是没有模板，而是模板存在、但没有进入真实协作链路。

---

## 9.8 最小练习

选一个最近完成的任务，倒推检查：

1. 当时有没有统一任务入口
2. review 时有没有固定检查项
3. 测试义务有没有提前说清楚
4. 复盘结论有没有反写进 Rules / Skills / Hooks

如果这 4 个问题中有两个以上答案是否定的，说明团队流程还停留在“个人经验”阶段。

# MCP

> **适合人群：** 工程师
> **学习目标：** 理解 MCP 的心智模型，知道它适合解决什么问题，能用最小配置把 AI 接入外部工具，同时守住权限和数据边界

---

## 5.1 本章解决什么问题

只靠聊天窗口里的文本，AI 很难真正参与真实工程流程。它不知道：

- GitHub 上的 issue 现在讨论到了哪一步
- 数据库 schema 长什么样
- 某个外部系统的实时状态是什么

MCP 解决的就是这个问题：**让 AI 从“只能读你贴进去的内容”变成“可以在受控边界内读取工具和资源”**。

---

## 5.2 为什么这件事重要

没有 MCP 时，人类经常充当“中转站”：

1. 手动去 GitHub / 数据库 / 浏览器查信息
2. 复制结果给 AI
3. 再把 AI 的分析拿回外部系统执行

MCP 的价值是把这条链路缩短，但前提是：

- 接入范围受控
- 权限最小
- 工具能力与任务风险匹配

所以 MCP 不只是“更方便”，它也是 AI 从单点问答走向真实工作流的关键一步。

---

## 5.3 核心概念与边界

### MCP 的三类能力

| 类型 | 作用 | 典型例子 |
|------|------|----------|
| Tools | 让 AI 调用外部工具 | 读 issue、查文件、查时间 |
| Resources | 让 AI 读取外部资源 | 文档、数据库 schema、配置 |
| Prompts | 提供预定义交互模板 | 固定的工作流 prompt |

### 一个简单心智模型

可以把 MCP 理解成“AI 可调用工具的统一协议层”。  
模型还是那个模型，只是它现在不必完全依赖你手工搬运上下文。

### 边界：不是所有能力都该开放

适合先接入 MCP 的，通常是：

- 只读类能力
- 查询类能力
- 风险低、容易验证的能力

不适合一开始就开放的，通常是：

- 直接写生产系统
- 删除、部署、批量修改外部资源
- 权限高但又缺少确认点的操作

## 5.4 一个最小场景与反例

### 正例：用 GitHub MCP 定位 issue 对应的代码入口

```
请读取 Issue #42，总结问题现象，再在代码库里找相关逻辑入口。
先给分析，不要直接修改。
```

这个场景适合 MCP，因为它主要是：

- 读取 issue
- 搜索代码
- 做分析

### 反例：一上来就给外部系统写权限

“既然已经能连 GitHub 了，那顺便开放自动创建分支、发评论、改 issue 状态吧。”

问题在于团队还没验证：

- 权限是否最小化
- 流程里有没有确认点
- AI 是否已经稳定理解任务上下文

MCP 的正确落地顺序，通常不是“先把能开的都开了”，而是“先只读，再逐步放开”。

---

## 5.5 实操方法：从最小可用配置开始

### 第一步：先选只读能力

最推荐的起点通常是：

- GitHub 读取 issue / PR / 代码
- 文件系统读取受限目录
- 时间、天气等低风险工具

### 第二步：用项目级配置而不是个人随意配置

项目级 `.claude/settings.json` 更适合团队共享，因为它至少能让：

- 接入范围可审查
- 配置差异可追踪
- 新成员复制成本更低

### 第三步：明确只读 / 读写边界

建议在接入文档里显式写出：

- 这个 MCP 当前开放到什么能力
- 哪些能力只读
- 哪些能力需要人工确认

### 第四步：为常见故障准备排查顺序

MCP 配好了但“不生效”，通常优先排查：

1. Token 是否正确注入
2. 配置文件路径是否放对
3. 服务是否真正启动
4. 工具名和调用方式是否一致
5. 权限是否不足

## 5.6 从模板到真实项目：替换时要改什么

以 <a href="/downloads/mcp/templates/mcp-config.json" download><code>mcp-config.json</code></a> 为例，真正落地到项目时通常至少要替换：

- 环境变量名
- 本地路径或工作目录
- 允许访问的资源范围
- MCP server 包版本（建议固定版本，按计划升级）
- 团队默认需要的 MCP 列表

如果你只是“照抄模板”，但没有改这些边界项，往往会得到一个看似能运行、实际上不适合团队直接使用的配置。

## 5.7 常见误区 / 风险提醒

| 误区 | 更稳妥的做法 |
|------|--------------|
| “能接越多工具越好” | 只接当前流程真正需要的能力 |
| “先开写权限，后面再管” | 先只读，再逐步放开并补确认点 |
| “模板能跑起来就算完成” | 还要校验路径、权限、环境变量和团队边界 |
| “MCP 是纯技术配置，不需要治理” | 一旦连上外部系统，它就天然涉及权限和流程控制 |

**风险提醒：**  
MCP 最大的风险不是配不起来，而是“配起来了，但没人明确知道它能做什么、不能做什么”。

## 5.8 最小练习

从下面三件事里任选一件，做一次最小接入：

1. 配一个只读 GitHub MCP，用它读取 issue
2. 配一个受限目录的文件系统 MCP，让 AI 读取固定目录
3. 打开最小示例项目的 `.claude/settings.json`，指出其中哪些字段落到真实项目时必须替换

练习目标不是接得多，而是把“能力边界”说清楚。

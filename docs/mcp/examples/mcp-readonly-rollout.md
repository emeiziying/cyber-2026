# MCP 从只读到受控扩权的最小落地步骤

## 背景

团队决定让 AI 助手通过 MCP 接入外部系统（代码仓库、项目管理工具、文档库等），但对"AI 能操作外部系统"这件事有顾虑——万一误操作怎么办？

本文用一个 4 步渐进方案，说明如何从零风险的只读接入开始，逐步扩展到受控的写入能力。

---

## 第一步：只读接入（第 1-2 周）

先只开放读取类能力，验证调用链路是否通畅：

| 能力 | 示例 | 风险 |
|------|------|------|
| 读取 issue / PR | 查看 PR 描述、评论、diff | 无 |
| 读取指定目录文件 | 浏览项目结构和源码 | 无 |
| 读取文档和 schema | 查阅 API 文档、数据库结构 | 无 |

**这一步的目标：** 验证 MCP 连接配置正确、上下文传递完整、AI 能基于外部信息给出有价值的回答。

**验证标准：**
- AI 能正确读取并引用 issue 内容
- 返回的信息与实际数据一致
- 没有超出只读范围的调用尝试

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "<read-only-token>"
      }
    }
  }
}
```

**注意：** Token 权限设为只读（`repo:read`），从配置层面杜绝误写。

---

## 第二步：补确认点（第 2-3 周）

在准备开放写能力之前，先写清楚规则，而不是先给权限再补规则：

### 必须回答的三个问题

1. **哪类操作需要人工确认？**
   - 创建 / 关闭 issue
   - 合并 PR
   - 修改仓库设置

2. **哪类仓库 / 目录允许被修改？**
   - 文档仓库：允许
   - 核心业务仓库：仅允许创建草稿 PR
   - 基础设施仓库：不允许

3. **Token 权限如何最小化？**
   - 每个 MCP Server 使用独立 Token
   - Token 权限严格匹配实际需要
   - 定期轮换，不使用个人 Token

### 写成文档

把这些规则写进项目级 `CLAUDE.md` 或团队 wiki，确保不是口头约定：

```markdown
## MCP 写权限规则
- 文档仓库（docs-*）：允许创建 PR
- 业务仓库（app-*）：仅允许创建草稿 PR，不允许直接推送
- 基础设施仓库（infra-*）：禁止任何写操作
- 所有写操作必须在执行前输出操作说明，等待人工确认
```

---

## 第三步：有限写入（第 3-4 周）

如果团队确实需要写能力，从低风险写入开始：

| 写入类型 | 风险等级 | 适合先开放 |
|----------|----------|-----------|
| 创建草稿评论 | 低 | ✅ |
| 更新非核心文档 | 低 | ✅ |
| 在测试环境写入临时数据 | 低 | ✅ |
| 创建草稿 PR | 低 | ✅ |
| 合并 PR | 高 | ❌ 暂不开放 |
| 修改仓库设置 | 高 | ❌ 暂不开放 |
| 删除资源 | 高 | ❌ 暂不开放 |

**实施要点：**
- 每次只增加一种写入能力，观察一周
- 出现一次误操作就暂停扩权，先补规则
- 用独立的测试仓库验证，不要直接在生产仓库试

---

## 第四步：保留回滚与审计（持续）

一旦 MCP 会写外部系统，就必须考虑可追溯性：

### 审计

- **谁触发了操作？** 记录操作者（人）和执行者（AI + MCP Server）
- **什么时间？** 所有写操作带时间戳
- **做了什么？** 记录完整的操作内容，不只是"调用了 API"

### 回滚

- **创建的资源能删除吗？** 草稿评论、草稿 PR 可以删除
- **修改的内容能恢复吗？** 文档修改有 git 历史可回溯
- **影响范围可控吗？** 如果 MCP 操作影响了下游系统，是否有通知机制

### 审批

对于高风险操作，考虑加一层审批：

```
AI 发起写操作请求 → 记录到审批队列 → 人工确认 → 执行
```

---

## 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 一开始就给全部权限 | 出了问题才发现权限太大 | 从只读开始，按需扩展 |
| 用个人 Token | 人员离职后 Token 无法管理 | 用团队 / 服务账号 Token |
| 没有测试环境先试 | 直接在生产系统上验证 | 先在测试仓库完整走一遍 |
| 规则只在口头约定 | 新人不知道边界在哪 | 写进 CLAUDE.md 或项目配置 |
| 一次开放多种写权限 | 出问题时不知道是哪个权限导致的 | 每次只加一种，观察后再扩展 |

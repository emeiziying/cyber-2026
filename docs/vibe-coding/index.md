# Vibe Coding

> **适合人群：** 工程师
> **学习目标：** 掌握 Vibe Coding 的基本方法，知道什么时候适合让 AI 高速推进，什么时候必须切回精确控制

---

## 2.1 本章解决什么问题

很多人第一次接触 AI 编程时，最常见的两个极端是：

- 要么什么都想让 AI 一次性做完
- 要么因为结果不稳定，最后又完全退回传统写法

本章解决的是中间地带：如何让 AI 在“高速度”和“可控性”之间工作得刚好合适。

---

## 2.2 为什么这件事重要

Vibe Coding 的价值不是“省掉思考”，而是**把思考从逐行实现转移到目标表达、快速迭代和结果验证**。

它最适合的场景通常具备这些特征：

- 目标明确，但实现细节可以探索
- 任务模式化，存在可复用套路
- 可以快速运行、快速观察、快速改
- 即使第一版不完美，也能很快继续迭代

如果任务本身无法快速验证，或者错误成本很高，Vibe 模式就不该成为默认做法。

---

## 2.3 核心概念与边界

### 什么是 Vibe Coding

Vibe Coding 的核心不是“随便说一句就让 AI 写代码”，而是：

1. 先说清目标
2. 用几轮对话快速逼近结果
3. 在关键节点要求 AI 解释、验证和收敛

### 一个简单公式

高质量输入通常包含这五部分：

```
目标 + 上下文 + 约束 + 输出形式 + 参考材料
```

### 边界：Vibe 模式不适合什么

下面几类任务不应该默认保持“纯 vibe”状态：

- 支付、权限、认证、安全逻辑
- 核心算法或高性能路径
- 一旦出错就很难发现的问题
- 无法快速运行和验证的复杂改动

这类任务可以让 AI 参与，但必须更早进入“精确控制 + 测试先行 + 逐步验证”模式。

---

## 2.4 一个最小场景与反例

### 正例：给现有项目补一个搜索接口

```
我的项目用 Node.js + Fastify + SQLite。
请先帮我实现一个搜索接口：
- 搜标题和内容
- 保持现有返回结构
- 不改已有 CRUD 行为
- 先给出你准备改哪些文件，再开始实现
```

这个输入有效，是因为它同时给出了：

- 目标：实现搜索接口
- 技术上下文：Node.js + Fastify + SQLite
- 边界：不改已有 CRUD 行为
- 过程约束：先列影响文件，再动手

### 反例：模糊而且无法验收

```
帮我把搜索做好一点
```

这类输入的主要问题不是短，而是完全没有告诉 AI：

- 什么叫“做好一点”
- 现有实现是什么
- 哪些行为不能变
- 做完后怎么判断是否真的更好

---

## 2.5 实操方法：四轮推进，而不是一步到位

### 第 1 轮：描述目标

只回答这件事要实现什么，不急着写实现细节。

### 第 2 轮：补约束

补技术栈、边界、参考文件、返回格式、不能碰的区域。

### 第 3 轮：让 AI 解释计划

在真正改动前，先让 AI 说清楚：

- 打算改哪些文件
- 为什么这么改
- 哪些地方最可能有风险

### 第 4 轮：要求验证和总结

完成后不要只看 diff，还要追问：

- 你做了哪些验证
- 哪些边界还没覆盖
- 如果我只 review 一处，最该看哪里

这种节奏比“一口气写完整功能”更稳，也更适合真实工程。

---

## 2.6 什么时候退出 Vibe 模式

出现下面任一信号，都应该切回更精确的控制方式：

1. AI 连续两到三轮都没有理解核心问题
2. 需求里包含权限、安全、兼容性或高风险边界
3. 你已经无法解释它生成的关键逻辑
4. 改动开始影响核心模块，却还没有测试保护
5. 你发现自己在不断“补一句再试一次”，但目标仍然不清楚

**结论：**  
退出 Vibe 模式不是失败，而是说明任务已经进入需要更高确定性的阶段。

---

## 2.7 常见误区 / 风险提醒

| 误区 | 更稳妥的做法 |
|------|--------------|
| “Prompt 越长越好” | 关键是信息完整，不是废话更多 |
| “第一版跑起来就算完成” | 必须追问边界、测试和未覆盖风险 |
| “AI 已经写了，所以我不需要理解” | 理解不了的关键逻辑，默认不能直接放行 |
| “所有任务都适合用同一种节奏” | 快速原型、修 bug、核心模块改造，节奏应该不同 |

**风险提醒：**  
Vibe Coding 最容易让人上头的地方，是前两轮看起来进展很快；真正的质量差异通常出现在第三轮以后的收敛和验证。

---

## 2.8 最小练习

任选你熟悉的一个小功能，按下面步骤做一次最小练习：

1. 用一句话描述目标
2. 再补上技术栈、边界、输出形式
3. 让 AI 先说改动计划，不直接写代码
4. 完成后追问“你验证了什么，还有什么没验证”

如果你能从这 4 步中看出输出质量明显提升，就已经掌握了 Vibe Coding 的基本节奏。

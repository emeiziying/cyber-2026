# 一个完整的 Vibe Coding 对话样例

## 场景

已有一个 Node.js + Fastify + SQLite 的笔记服务（就是 Minimal Agent Demo 中的项目），现在要给它加一个搜索功能。

这个样例展示了从"模糊需求"到"可验证实现"的完整 4 轮对话过程。

---

## 起点：模糊需求（反面示例）

> 帮我加一个搜索功能。

这个说法太短，AI 很难知道：
- 搜索什么？标题？内容？标签？
- 范围多大？只改后端还是前后端都改？
- 已有行为能不能变？
- 怎么算"做完了"？

**不要从这里开始。** 下面是正确的做法。

---

## 第一轮：提供上下文和约束

> 我有一个 Node.js + Fastify + SQLite 的笔记服务。
> 请帮我加一个搜索接口，支持标题和内容模糊搜索。
> 不要改已有 CRUD 行为。
> 先告诉我你准备改哪些文件、为什么这么改。

### AI 应该输出什么

- 需要修改的文件清单（如 `note-repository.js`、`server.js`）
- 每个文件要做什么改动
- 为什么选择这个方案（比如"用 SQLite 的 LIKE 而不是引入全文索引，因为数据量小"）
- 明确说明不会改什么

### 你要检查什么

- [ ] 改动文件范围是否合理？有没有不该改的文件？
- [ ] 方案是否过度设计？（小项目不需要 Elasticsearch）
- [ ] 是否理解了"不改已有 CRUD"这个约束？

---

## 第二轮：补充细节和验证标准

> 搜索结果要保持现有返回结构（和 GET /notes 一致）。
> 如果没有结果，返回空数组，不要报错。
> 搜索关键词为空时，返回全部笔记。
> 再补充一下你准备怎么验证。

### 为什么需要这一轮

第一轮定了方向，但还有很多细节没有明确。如果不补充，AI 可能会：
- 给搜索结果用一个全新的返回格式 → 前端要改两套解析逻辑
- 没有结果时返回 404 → 前端会当成错误处理
- 空关键词时报 400 → 和"返回全部"的预期不一致

### AI 应该输出什么

- 确认返回格式与现有接口一致
- 边界情况的处理策略（空关键词、特殊字符、超长输入）
- 验证计划（手动测试步骤或自动化测试方案）

---

## 第三轮：执行实现

> 现在先实现。实现后请：
> 1. 列出你改了哪些文件
> 2. 说明可能遗漏的边界
> 3. 给我一组最小测试建议

### AI 应该输出什么

**改动清单：**
- `src/repositories/note-repository.js` — 新增 `search(keyword)` 方法
- `src/server.js` — 新增 `GET /notes/search?q=keyword` 路由
- `test/note-service.test.js` — 新增搜索相关测试

**可能遗漏的边界：**
- SQL 注入风险（是否用了参数化查询？）
- 搜索关键词含 `%` 或 `_` 等 SQLite 通配符
- 大量数据时的性能（当前没有分页）

**最小测试建议：**
- 搜索存在的关键词 → 返回匹配结果
- 搜索不存在的关键词 → 返回空数组
- 空关键词 → 返回全部笔记
- 特殊字符 → 不报错，返回合理结果

---

## 第四轮：验收和收尾

> 运行测试，确认全部通过。然后：
> 1. 确认现有 CRUD 测试没有被破坏
> 2. 总结这次改动的范围和风险

### 验收检查清单

- [ ] 新增测试全部通过
- [ ] 已有测试全部通过（回归无破坏）
- [ ] 搜索接口返回格式与现有接口一致
- [ ] 边界情况（空、特殊字符）行为正确
- [ ] 没有改动不在范围内的文件

---

## 什么时候该停下来

如果 AI 在两轮之后仍然没有说清以下三件事，**不要继续"再试一轮"**：

1. 会改哪些文件
2. 为什么这样改
3. 如何验证没有破坏现有行为

这时应该先回头把目标和边界讲清楚，而不是寄希望于"多聊几轮就对了"。

---

## 对话质量的自检信号

| 信号 | 含义 | 行动 |
|------|------|------|
| AI 反复问你"你想要什么" | 你的输入缺少关键信息 | 补充上下文和约束 |
| AI 改了你没提到的文件 | 范围没有明确约束 | 加一条"不要改 XX" |
| AI 一次性输出大量代码但没有说明 | 你没有要求分步执行 | 回到第一轮，先要方案再要实现 |
| AI 的方案和你预期完全不同 | 可能是你的约束有歧义 | 用更具体的语言重新描述 |
| 测试覆盖了 happy path 但没有边界 | 你没有明确边界场景 | 在第二轮补充边界条件 |

---

## 小结

Vibe Coding 不是"随便说说让 AI 写"，而是一套**结构化的对话方法**：

1. **第一轮**：给上下文、定方向、要方案
2. **第二轮**：补细节、定边界、要验证计划
3. **第三轮**：要实现、要改动清单、要遗漏说明
4. **第四轮**：要测试、要回归确认、要风险总结

每一轮的输出都是下一轮的输入。跳过任何一步，后面的质量都会下降。

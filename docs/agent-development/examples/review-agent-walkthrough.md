# 代码审查 Agent 的最小设计闭环

## 背景

团队希望在 PR 提交后，自动完成一轮初步代码审查，把明显问题提前暴露，减轻人工 reviewer 的负担。

目标不是替代人工 review，而是让人把精力集中在架构判断和业务逻辑上，而不是格式、缺失测试、明显边界遗漏这类机械检查。

---

## 目标定义：这个 Agent 要做什么

读取一个 PR，优先找出三类问题：

1. **高风险文件变更**——是否改了认证、支付、权限等核心模块
2. **测试缺失**——新增或修改的逻辑是否有对应测试
3. **明显边界遗漏**——空值处理、错误码返回、并发场景等

输出一份结构化的 review finding，而不是一段笼统的"代码看起来不错"。

---

## 最小工具集

| 工具 | 用途 | 权限 |
|------|------|------|
| 读取 PR diff | 获取变更文件列表和具体改动 | 只读 |
| 搜索相关文件 | 查找被改动模块的上下文依赖 | 只读 |
| 读取测试文件 | 确认是否有对应测试覆盖 | 只读 |

**关键原则：** 不要一开始就给十几种工具。先保证 Agent 能围绕一个目标稳定循环，再考虑扩展。写权限在这个阶段完全不需要。

---

## 执行流程设计

```
┌─────────────────┐
│  读取 PR diff    │
└────────┬────────┘
         ▼
┌─────────────────┐
│  分类变更文件    │ ← 是否涉及高风险目录？
└────────┬────────┘
         ▼
┌─────────────────┐
│  检查测试覆盖    │ ← 新增逻辑是否有对应测试？
└────────┬────────┘
         ▼
┌─────────────────┐
│  扫描边界条件    │ ← 空值、错误码、并发处理
└────────┬────────┘
         ▼
┌─────────────────┐
│  生成结构化报告  │
└─────────────────┘
```

---

## 停止条件

满足下面任一条件就停止，不要无限循环：

- **正常完成：** 已经形成结构化 review finding，覆盖了三类检查目标
- **信息不足：** 缺少关键上下文（如依赖的私有库文档），需要人工补充
- **风险过高：** 发现改动涉及核心安全模块且缺乏测试，应标记风险后交还给人
- **超出能力：** 涉及复杂业务逻辑判断（如定价策略是否正确），不在 Agent 能力范围内

---

## 输出格式示例

```markdown
## PR Review Finding

### 高风险变更
- `src/auth/token-refresh.js` — 修改了 token 刷新逻辑，属于认证核心链路

### 测试缺失
- `src/services/order-service.js` 新增了退款计算方法，未找到对应测试文件
- 建议补充：正常退款、部分退款、超额退款三种边界

### 边界检查
- `src/api/search.js:42` — 搜索关键词为空字符串时未做处理
- `src/api/user.js:87` — 删除用户时未检查是否有关联的未完成订单

### 总结
- 风险等级：中
- 建议：补充测试后再合并，认证链路改动需要 Senior 确认
```

## 常见失败模式

| 失败 | 原因 | 对策 |
|------|------|------|
| 一开始就给写权限 | 误以为 Agent 需要写评论才能工作 | Review Agent 只需要读，输出交给人决定是否发布 |
| 没有定义"什么叫高风险文件" | Agent 不知道哪些目录重要 | 在 Rules 中明确列出高风险目录清单 |
| 找到问题后不停止 | 没有定义退出条件，Agent 反复扫描 | 明确停止条件和最大循环次数 |
| 输出太笼统 | 只说"建议优化"但不指向具体文件和行号 | 要求输出必须包含文件路径和行号 |
| 试图理解业务正确性 | Agent 判断"这个定价逻辑对不对" | 把业务判断明确标记为超出范围，交给人 |

---

## 从这个 Agent 到下一步

当这个最小 Agent 稳定运行后，可以逐步扩展：

1. **第一步：** 把输出接入 PR 评论（增加写权限，但仅限评论）
2. **第二步：** 添加项目特有的规则检查（如"不允许在 controller 层直接查数据库"）
3. **第三步：** 接入 CI 流水线，自动触发

但每一步都要先验证上一步是否稳定，不要跳着来。

# Agent 开发

> **适合人群：** 工程师（已具备基础 AI 工具使用经验）
> **学习目标：** 理解 Agent 的工作方式、失败边界和适用场景，知道什么时候应该自己设计简单 Agent，什么时候只用现成工具就够了

---

## 7.1 本章解决什么问题

学到前面几章后，很多人都会开始问两个问题：

- 既然 Claude Code 这类工具已经像 Agent 了，我还需要自己开发 Agent 吗？
- 什么任务值得做成单 Agent 或多 Agent，什么任务其实只是普通自动化？

本章解决的不是“如何堆一个炫技 demo”，也不是在这里实现某个业务 Agent，而是**如何判断 Agent 真正有价值的边界**。

---

## 7.2 为什么这件事重要

Agent 的价值不在于“它能多想几步”，而在于它能把以下动作串成闭环：

- 理解目标
- 规划步骤
- 调用工具
- 观察结果
- 决定下一步
- 在合适时停下来并交回给人

如果一个任务本来只需要一个脚本、一个 Skill 或一个 MCP 查询，就没必要硬做成 Agent。  
Agent 真正适合的是**需要多轮判断、依赖外部观察、步骤之间有反馈关系的任务**。

---

## 7.3 核心概念与边界

### Agent 与普通对话式工具的区别

| 维度 | 对话式工具 | Agent |
|------|------------|-------|
| 主要模式 | 你问一句，我答一句 | 你给目标，我尝试完成任务 |
| 工具使用 | 通常由人显式发起 | 可以根据状态自主决定 |
| 多步骤推进 | 主要靠人串联 | 可以自己循环执行 |
| 停止条件 | 人类决定结束 | 达到目标、失败或遇到边界时停止 |

### 一个最小心智模型

```
目标 → 规划 → 调工具 → 观察结果 → 判断是否继续 → 返回结果
```

### 边界说明

本套方法论在这一部分提供的是 Agent 的适用性判断、最小 loop 设计和失败边界分析，不承载真实专项 Agent 的代码实现。
如果某个 Agent 最终值得落地，真实实现通常应放在独立工程或具体业务仓中推进。

### 边界：不是所有自动化都该叫 Agent

下面这些更像脚本或 Skill，而不是 Agent：

- 固定输入、固定输出、没有中途分支的任务
- 单次调用就能完成的转换
- 几乎不需要外部观察或反馈判断的流程

如果任务本身没有循环、判断和多步反馈，就没必要引入 Agent 这个复杂度。

---

## 7.4 一个最小场景与反例

### 正例：代码审查 Agent

“读取 PR 变更，先找高风险文件，再检查测试覆盖，最后输出结构化 review finding。”

这个场景适合 Agent，因为它天然需要：

- 读取多类上下文
- 根据观察结果决定下一步
- 在多个候选风险里收敛重点

### 反例：把单次格式转换硬包装成 Agent

“把这段 JSON 转成表格 markdown。”

这类任务更适合普通 prompt 或 Skill，因为它没有明显的工具循环和状态判断。  
如果硬做成 Agent，只会增加复杂度和维护成本。

---

## 7.5 实操方法：从单 Agent 到多 Agent 的渐进路径

### 第一步：先用现成 Agent 完成复杂任务

在真正自己开发前，先回答：

- 现有工具是否已经能完成 80% 的效果
- 团队是否真的需要自定义循环和工具编排

### 第二步：做最小单 Agent

第一版自定义 Agent 最适合解决：

- 输入来源明确
- 工具数量少
- 停止条件清楚
- 结果能快速验证

这一步优先输出最小设计闭环，不要求在这里直接承载真实专项实现。

### 第三步：再考虑 Sub-Agent 或 Multi-Agent

只有当任务可以**明确拆成相互独立的子问题**时，才值得引入多 Agent，例如：

- 一个子 Agent 扫测试覆盖
- 一个子 Agent 查安全问题
- 一个子 Agent 查依赖漏洞

如果拆分后彼此高度耦合、经常互相等待，说明还不到多 Agent 的时机。

---

## 7.6 Agent 的失败模式

下面这些失败模式在实际开发里最常见：

1. **循环太长，目标越来越漂**  
   一开始目标不清晰，Agent 越做越偏。

2. **工具太多，权限太大**  
   结果是调试成本和风险都上升。

3. **停止条件不清楚**  
   Agent 不知道什么时候该交还给人。

4. **把并行当成价值本身**  
   多 Agent 并不自动等于更快；错误拆分只会带来更多整合成本。

5. **没有验证闭环**  
   Agent 能做很多步，但没人定义“什么叫做完”。

**结论：**  
Agent 的核心难点不是生成，而是边界、停机点和验证。

## 7.7 常见误区 / 风险提醒

| 误区 | 更稳妥的做法 |
|------|--------------|
| “能做成 Agent 的都应该做成 Agent” | 先判断 Skill、脚本或现成工具是否已经足够 |
| “多 Agent 一定比单 Agent 更先进” | 只有任务可明确拆分时，多 Agent 才有收益 |
| “Agent 自主性越高越好” | 高自主性必须配套更强的权限边界和停机条件 |
| “只要效果看起来对，就算 Agent 可用” | 必须定义失败模式、验证方法和人工接管点 |

**风险提醒：**  
Agent 最容易被高估的地方，是“演示时看起来很聪明”；真正决定可用性的，往往是失败时能否安全停下来。

---

## 7.8 最小练习

选一个你熟悉的多步骤任务，先不要写代码，先用下面 4 个问题判断它是否适合做成 Agent：

1. 这个任务是否需要多轮观察和判断？
2. 是否依赖多个工具或上下文来源？
3. 停止条件是否明确？
4. 出错后是否容易验证和人工接管？

如果这 4 个问题里有 2 个以上答不上来，通常说明它还不适合直接进入 Agent 开发。

# Skill 与 Hook 的选择示例

## 核心判断原则

在决定用 Skill 还是 Hook 之前，先回答一个问题：

> 这件事是**用户主动发起**的，还是**某个事件发生时应该自动执行**的？

- 主动发起 → **Skill**（用户用 `/command` 触发）
- 自动执行 → **Hook**（系统在特定事件时自动运行）

---

## 场景 1：统一 bug 修复输出格式

**情况：** 团队发现不同人让 AI 修 bug 时，输出格式差异很大——有人只改代码不写说明，有人补了测试但没列影响范围。团队希望统一修复流程。

**更适合：** Skill

**原因：**
- 修 bug 是用户主动发起的任务，不是自动触发的
- 需要固定的输入提示（问题描述、复现步骤）和输出格式（根因、修复、测试、影响）
- 不同 bug 可能需要不同的处理深度，用户需要控制

**实现方式：** 创建 `.claude/commands/fix-bug.md`，定义完整的修复流程模板。

---

## 场景 2：禁止在 main 分支直接写文件

**情况：** 有人在 main 分支上直接让 AI 修改文件并提交，绕过了 PR 流程。

**更适合：** Hook

**原因：**
- 这类约束应该自动触发，而不是靠用户记得先运行一个命令
- 它是一种**保护机制**，必须在每次文件写入时检查
- 用户不应该有"选择跳过"的机会

**实现方式：** 在 `.claude/settings.json` 中配置 PreToolUse Hook，在文件写入前检查当前分支：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "if [ \"$(git branch --show-current)\" = 'main' ]; then echo 'BLOCK: 不允许在 main 分支直接修改文件，请先切换到功能分支'; exit 1; fi"
          }
        ]
      }
    ]
  }
}
```

## 场景 3：会话开始时自动读取项目环境信息

**情况：** 每次开新会话，AI 都不知道当前项目用什么框架、什么包管理器、什么测试工具，需要人重新解释。

**更适合：** Hook

**原因：**
- 这是稳定、重复、无需人工决策的初始化动作
- 每次会话开始时都该执行，不应该依赖用户记得手动触发
- 收集的信息是客观事实（框架版本、依赖列表），不需要人工判断

**实现方式：** 配置 SessionStart Hook，自动读取 `package.json`、检测框架版本、列出关键目录结构。

## 场景 4：生成日报

**情况：** 团队要求每天结束时生成一份当日工作总结，包含完成的任务、提交的 PR、遇到的问题。

**更适合：** Skill

**原因：**
- 它是一个有固定输入和固定输出格式的主动任务
- 用户可能想在不同时间点触发（下班前、会议前）
- 日报内容可能需要用户补充上下文（比如"今天主要在处理 XX 需求"）

**实现方式：** 创建 `.claude/commands/daily-report.md`，自动收集当日 git log 和 PR 状态，结合用户补充生成结构化日报。

---

## 场景 5：代码提交前自动运行 lint

**情况：** 团队约定所有代码在提交前必须通过 lint 检查，但有人忘记手动运行。

**更适合：** Hook

**原因：**
- 这是提交事件发生时应该自动执行的检查
- 不需要用户决策（lint 规则已经确定）
- 遗忘的成本高（不合格代码进入仓库）

**实现方式：** 配置 PreToolUse Hook，在 `Bash` 工具执行 `git commit` 时自动先运行 `pnpm lint`。

---

## 场景 6：生成团队 Rules

**情况：** Tech Lead 希望基于项目现有代码风格和约定，自动生成一份项目级 CLAUDE.md 草稿。

**更适合：** Skill

**原因：**
- 这是一次性的主动任务，不是每次都要执行的
- 需要分析大量代码和配置，输出一份完整文档
- 生成结果需要人工审核和调整

---

## 快速决策表

| 特征 | Skill | Hook |
|------|-------|------|
| 触发方式 | 用户主动（`/command`） | 事件自动触发 |
| 是否需要用户输入 | 通常需要 | 通常不需要 |
| 执行频率 | 按需 | 每次事件发生 |
| 是否可跳过 | 用户可以选择不用 | 不应该被跳过 |
| 典型用途 | 生成、分析、报告 | 检查、保护、初始化 |

---

## 常见误判

| 误判 | 问题 | 修正 |
|------|------|------|
| 把保护机制做成 Skill | 用户可以选择不运行，保护形同虚设 | 改为 Hook |
| 把复杂生成任务做成 Hook | 每次事件都触发大量计算，拖慢工作流 | 改为 Skill |
| 一个 Skill 里塞太多步骤 | 用户不知道跑了什么、为什么慢 | 拆成多个小 Skill |
| Hook 里做需要判断的操作 | Hook 没有机会让用户确认就执行了 | 只做客观检查，判断交给 Skill |

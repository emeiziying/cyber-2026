# Rules

> **适合人群：** 工程师
> **学习目标：** 通过项目级规则文件让 AI 真正理解你的项目上下文，减少重复纠正，并把团队隐性规范沉淀成可共享资产

---

## 4.1 本章解决什么问题

很多团队第一次把 AI 接进项目时，都会出现同一种低效：

- 每次都要重新解释技术栈
- 每次都要提醒命名规范和目录结构
- 每次都要阻止 AI 去碰一些明确不该碰的区域

这说明问题不在模型本身，而在于项目知识没有被显式写出来。Rules 的作用，就是把这些原本散落在口头习惯、代码风格和 review 评论里的经验，提前交给 AI。

---

## 4.2 为什么这件事重要

Rules 不是“写给 AI 看的一份提示词”，而是**把项目约束前置的一层工程配置**。

它的价值主要有三点：

1. **减少重复纠正**  
   同一类技术栈、风格和边界不需要每次重说。

2. **让个人经验变成团队资产**  
   不再依赖“谁更会跟 AI 说话”，而是团队共享同一套约束。

3. **把 review 问题前移**  
   很多本来要在 review 阶段纠正的问题，可以在 AI 动手前就被约束住。

如果没有 Rules，AI 的输出是否稳定，更多取决于单次对话质量；有了 Rules，稳定性才开始像工程能力，而不是聊天技巧。

---

## 4.3 核心概念与边界

### Rules 的常见位置与优先级

| 工具 | 典型位置 | 适合存放什么 |
|------|----------|--------------|
| Claude Code | 项目根目录 `CLAUDE.md` | 团队共享规则 |
| Claude Code | 子目录 `CLAUDE.md` | 模块级特殊约束 |
| Claude Code | `~/.claude/CLAUDE.md` | 个人偏好 |
| Cursor | 项目根目录 `.cursorrules` | IDE 项目级规则 |

**优先级原则：** 越接近目标代码的位置，规则越具体，优先级越高。

### Rules 应该解决什么，不应该解决什么

**适合写进 Rules：**

- 技术栈与框架约束
- 目录结构和命名规范
- 禁止行为
- 运行 / 测试 / 构建命令
- review 重点和安全边界

**不适合写进 Rules：**

- 临时一次性的任务指令
- 敏感 token、账号、私钥
- 过度细碎、经常变化的琐碎偏好

Rules 的目标是形成稳定边界，不是把每次对话都塞进一份永久文件。

---

## 4.4 一个最小场景与反例

### 反例：没有 Rules 时的典型对话

```
你：帮我写一个 React 组件
AI：用了 class 组件和 CSS Modules
你：我们不用 class 组件
AI：改成函数组件
你：我们也不用局部 state，统一用 Zustand
AI：好的……
```

这种对话的问题不是 AI 不聪明，而是项目约束根本没有被提前声明。

### 正例：有 Rules 时的约束表达

```markdown
## 前端约束
- 使用函数组件和 Hooks
- 状态管理统一使用 Zustand
- 样式统一使用 Tailwind CSS
- 新增组件时优先查看现有 `components/` 是否已有可复用实现
```

**结论：**  
好的 Rules 不是为了让 AI 听话，而是为了让它一开始就站在正确上下文里工作。

## 4.5 实操方法：把隐性规范变成显性规则

### 最小可用 Rules 建议包含 5 部分

1. 项目是什么
2. 技术栈与主要目录
3. 必须遵守的代码规范
4. 明确禁止的行为
5. 常用命令与验证方式

### 坏 Rules vs 好 Rules

| 写法 | 问题 | 更好的写法 |
|------|------|------------|
| “代码要清晰” | 太抽象，AI 无法执行 | “函数超过 50 行时优先拆分，并提取纯函数” |
| “注意安全” | 无法落地 | “不要把内部异常信息返回给用户；统一返回通用错误和错误 ID” |
| “遵循项目风格” | 风格未定义 | “返回结构统一为 `{ data, error }`，失败时 `data` 为 `null`” |

### 一个推荐的演进流程

1. 先写最小版本，不求完整
2. 当 AI 反复犯同一类错误时，再补对应规则
3. 当目录结构或技术栈变化时，同步更新
4. 定期删掉过时、冲突或没人理解的规则

Rules 不是一次写完的，它更像一份随着项目演进的“AI 使用约束”。

## 4.6 真实冲突场景：全局规则、项目规则、子目录规则

一个常见问题是：当多个层级的规则同时存在时，谁说了算？

**示例：**

- 全局规则：默认使用英文注释
- 项目规则：公共 API 说明必须用中文
- `src/security/CLAUDE.md`：安全模块禁止新增注释，只允许补文档到专门目录

在这种情况下：

- 处理安全模块文件时，优先遵守安全模块的子目录规则
- 处理其他公共 API 文件时，遵守项目规则
- 全局规则只在没有更具体规则时兜底

如果团队经常出现类似冲突，说明应该补一份“优先级说明”或拆分规则职责。

---

## 4.7 常见误区 / 风险提醒

| 误区 | 更稳妥的做法 |
|------|--------------|
| “Rules 越长越好” | 重点是高频约束和清晰边界，不是堆字数 |
| “只写技术栈就够了” | 还要写禁止行为、验证命令和 review 重点 |
| “个人偏好都写进项目规则” | 项目规则放团队共识，个人偏好放全局规则 |
| “Rules 写完就不用管了” | 随技术栈和 review 问题持续演进 |

**风险提醒：**  
Rules 最容易失效的原因不是没人写，而是没人维护，最后变成一份大家都不再相信的文档。

## 4.8 最小练习

从你手上的一个真实项目里，先补这 5 条最小规则：

1. 项目技术栈
2. 目录结构里的 2-3 个关键目录说明
3. 至少 3 条禁止行为
4. 测试 / 构建 / 启动命令
5. 一条最容易在 review 里反复出现的提醒

只要这 5 条写得具体，AI 输出就会明显稳定。

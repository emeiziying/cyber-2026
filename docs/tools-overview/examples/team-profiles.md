# 常见团队画像与工具组合

不同团队的技术水平、管理要求和工作模式差异很大。这里列出 4 种常见画像，帮助团队找到适合自己的起步方式，而不是盲目照搬别人的工具组合。

---

## 画像 1：VS Code 重度团队，想低成本起步

### 特征

- 全员使用 VS Code，不想更换 IDE
- 以日常编码补全和内联建议为主
- 暂时不需要复杂自动化，先解决"写代码更快"
- 团队 3-8 人，没有专职的工具链维护者

### 推荐组合

| 层级 | 工具 | 用途 |
|------|------|------|
| 第一层 | IDE 内补全（Copilot / Cursor） | 日常编码的默认入口 |
| 第二层 | 共享 Rules（CLAUDE.md） | 统一 AI 行为约束 |
| 第三层（后续） | CLI Agent | 处理跨文件修改、批量重构 |

### 起步路径

1. **第 1 周：** 全员安装 IDE 插件，自由使用
2. **第 2 周：** 收集"AI 做错了什么"的共性问题，写成 5 条基础 Rules
3. **第 3 周：** 把最高频的任务（如修 bug）做成简单的提示词模板
4. **第 4 周后：** 评估是否需要 CLI 工具处理更复杂的任务

### 注意事项

- 不要一开始就上 CLI Agent——如果团队连 Rules 都没有，CLI 的灵活性反而会放大混乱
- IDE 插件已经能覆盖 70% 的日常场景，不要低估它的价值

---

## 画像 2：已有较强工程规范，想推动复杂任务自动化

### 特征

- 团队有成熟的 CI/CD、Code Review 流程
- 成员愿意接受 CLI 工作流，不排斥终端操作
- 需要处理跨文件修改、批量测试、Review 辅助等复杂任务
- 有能力维护规则和模板，通常有 Tech Lead 或平台工程师

### 推荐组合

| 层级 | 工具 | 用途 |
|------|------|------|
| 核心 | Agent 类 CLI 工具（Claude Code） | 复杂任务的主入口 |
| 配套 | 项目级 Rules + Skills | 约束 AI 行为，统一任务流程 |
| 扩展 | 最小 MCP 配置 | 接入 GitHub、Jira 等外部系统 |
| 辅助 | IDE 插件 | 简单编码补全 |

### 起步路径

1. **第 1 周：** 写好项目级 CLAUDE.md，定义 10-15 条核心规则
2. **第 2 周：** 创建 3-5 个常用 Skill（fix-bug、gen-tests、review-code）
3. **第 3 周：** 配置只读 MCP 接入（读取 issue 和 PR 信息）
4. **第 4 周后：** 根据需要扩展 MCP 写权限、增加 Hook 保护

### 注意事项

- 规则和 Skill 应该纳入版本管理，像对待代码一样对待它们
- MCP 扩权要渐进，参考 [MCP 从只读到受控扩权](../../mcp/examples/mcp-readonly-rollout)

---

## 画像 3：管理要求严格，优先考虑数据与权限边界

### 特征

- 对代码上传、第三方连接、Token 权限有严格合规要求
- 行业可能涉及金融、医疗、政府等受监管领域
- 更关注可控性和可审计性，而不是"用 AI 用得爽"
- 审批流程较长，新工具引入需要多方评估

### 推荐组合

| 层级 | 工具 | 用途 |
|------|------|------|
| 核心 | 边界清晰的最小能力集 | 只开放必要的功能 |
| 原则 | 只读优先 | 先证明安全，再讨论效率 |
| 延后 | 高自主能力工具 | 等团队建立治理框架后再引入 |

### 起步路径

1. **第 1-2 周：** 评估哪些工具满足数据驻留和审计要求
2. **第 3-4 周：** 在隔离环境中试点，只读模式
3. **第 5-6 周：** 制定 AI 使用的合规清单和审计日志方案
4. **第 7 周后：** 在受控范围内逐步扩展

### 注意事项

- 不要因为合规要求高就完全不引入——这样会被竞争对手拉开差距
- 可以先在文档、测试等低风险场景试点，积累经验后再扩展到核心业务
- 合规清单应该由安全团队和工程团队共同制定，不要只让一方决定

## 画像 4：混合团队，需要兼顾产品、设计和研发

### 特征

- 不只是程序员使用 AI，产品经理、设计师也需要参与
- 需要统一输入格式、验收标准和协作方式
- 不同角色对 AI 的理解水平差异大
- 通常 10 人以上，跨部门协作

### 推荐组合

| 角色 | 工具 | 用途 |
|------|------|------|
| 研发 | CLI Agent + IDE 插件 | 编码和工程任务 |
| 产品 / 设计 | 跨角色 AI 手册 + 任务模板 | 统一方法论和输入格式 |
| 全员 | 共享规则 + 验收清单 | 保证协作一致性 |

### 起步路径

1. **第 1 周：** 全员阅读[跨角色 AI 手册](../../appendix-case-studies/cross-functional-ai-handbook)，建立共同语言
2. **第 2 周：** 研发侧配置工程工具，产品 / 设计侧试用 AI 辅助文案和需求撰写
3. **第 3 周：** 制定跨角色的任务入口模板（统一"怎么给 AI 提需求"）
4. **第 4 周后：** 定期复盘，把有效经验写入团队级文档

### 注意事项

- 非研发角色不需要学 CLI——给他们准备的是阅读材料、模板和培训，不是工具
- 最大的风险不是技术，而是"研发觉得很好用但其他人不知道怎么用"的信息差
- 建议在正式推广前先做一次 [30 分钟 Workshop](../../presentation-outlines/30min-cross-functional-ai-workshop)

## 怎么判断自己属于哪种画像

| 问题 | 画像 1 | 画像 2 | 画像 3 | 画像 4 |
|------|--------|--------|--------|--------|
| 团队是否接受 CLI？ | 不太 | 接受 | 视情况 | 部分接受 |
| 是否有严格合规要求？ | 没有 | 一般 | 严格 | 一般 |
| 是否只有研发使用？ | 是 | 是 | 是 | 不是 |
| 是否有工具链维护者？ | 没有 | 有 | 有 | 可能有 |
| 最急迫的需求？ | 写代码更快 | 复杂任务自动化 | 安全可控 | 跨角色协作 |

找到最接近的画像后，按对应的起步路径开始。不需要一步到位，每个画像都是从最小可行方案开始，逐步扩展。

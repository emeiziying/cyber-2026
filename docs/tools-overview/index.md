# 工具全景

> **适合人群：** 全员（工程师 + 技术管理者）
> **学习目标：** 了解主流工具的差异，不只会比较功能，还能根据团队约束和使用场景做选型判断

---

## 3.1 本章解决什么问题

知道工具名字，不等于会选工具。团队在选型时最常见的问题不是信息太少，而是评价维度太单一：

- 只看“谁更强”，不看团队怎么用
- 只看单人体验，不看协作和治理
- 只看功能，不看采购、权限、数据边界和迁移成本

本章要解决的是：**怎样把“我觉得好用”转换成“团队适合用”**。

---

## 3.2 为什么工具选型不能只看功能

一个 AI 编程工具是否适合团队，至少要同时回答下面几类问题：

- 它嵌入现有工作流的成本高不高
- 是否支持项目级规则、模板和共享约束
- 它能不能处理复杂任务，还是只适合补全
- 数据边界、权限模型和企业采购是否可接受
- 团队是否愿意围绕它统一一套方法

所以真正有用的比较方式不是“谁最聪明”，而是“谁最适合当前阶段的团队”。

## 3.3 核心概念与边界

### 按使用形态分类

| 类别 | 代表工具 | 典型优势 | 典型限制 |
|------|----------|----------|----------|
| IDE 原生类 | Cursor、Copilot | 上手快、贴近日常编码 | 自动化和外部工具接入较有限 |
| CLI / Agent 类 | Claude Code、Aider | 擅长复杂任务、跨文件处理、自动化 | 需要更强的命令行和流程意识 |
| Web / Chat 类 | Claude Web、ChatGPT | 轻量问答、探索、无代码上下文讨论 | 缺少项目上下文，难做直接落地 |
| 平台 / 编排类 | OpenClaw | 常驻接入消息平台、多 Agent 协作、跨工具编排、持久记忆 | 初始配置成本高、需要服务器运维能力 |

### 边界：没有“唯一正确工具”

工具选型不是一次性终局决策。更常见的成熟做法是：

- 用 IDE 工具承接日常编码补全
- 用 CLI / Agent 工具处理复杂任务和批量工作
- 用 Web 工具做轻量探索和非代码问答

也就是说，团队真正需要的是**组合策略**，而不是宗教战争。

## 3.4 一个最小判断场景

**场景：** 团队 8 个人，主要做中后台应用，成员都长期用 VS Code，短期内不想更换 IDE，但又想开始做多文件改动和自动化 Review。

### 容易出现的误判

“既然 Claude Code 的 Agent 能力最强，那就要求所有人都切过去。”

这类决定的问题在于忽略了：

- IDE 迁移成本
- 团队现有工作习惯
- 不同角色的学习曲线
- 某些成员只需要补全，不需要重型自动化

### 更稳妥的判断

- 保留现有 IDE 生态中的轻量工具
- 给复杂任务和批量任务引入 CLI Agent
- 统一共享规则和验收方式，而不是强制所有人用同一个入口

工具统一不是目的，**协作方式统一**才是目的。

## 3.5 实操方法：五个选型维度

### 1. 任务复杂度

如果团队常见任务是跨多个文件、需要分析代码库和执行多步骤操作，Agent 类工具价值更大。

### 2. 团队迁移成本

如果团队严重依赖现有 IDE 插件、快捷键和调试链路，强行切 IDE 的收益未必大于成本。

### 3. 共享规则能力

项目级 Rules、模板、Skills、Hooks 能否共享，决定了工具能不能从“个人效率工具”升级为“团队能力载体”。

### 4. 数据与权限边界

采购、隐私、代码上传范围、第三方连接能力，这些往往比模型效果更能决定企业是否能真正落地。

### 5. 治理和自动化

如果你希望 AI 不只是补全，而是能接 GitHub、文件系统、测试和 Review 流程，就必须看工具的外部工具调用能力。

## 3.6 主流工具横向对比

| 维度 | Claude Code | Cursor | GitHub Copilot | Aider | OpenClaw |
|------|-------------|--------|----------------|-------|----------|
| 形态 | CLI / Agent | IDE | IDE 插件 | CLI | 持久化守护进程 + 消息平台 |
| 典型优势 | 复杂任务、自主执行、外部工具 | IDE 体验完整、迁移顺滑 | 补全和 IDE 集成成熟 | 终端工作流强、开源友好 | 消息平台接入、多 Agent 并行、跨工具编排 |
| 典型限制 | 更依赖命令行习惯 | 团队治理能力取决于使用方式 | 自主多步骤能力相对弱 | 企业级治理和生态整合较弱 | 初始配置成本高、需服务器运维 |
| 更适合谁 | 需要自动化和复杂任务的团队 | IDE 重度用户 | 想低成本叠加 AI 的团队 | 终端重度用户和个人开发者 | 需要团队机器人和跨工具自动化的团队 |

## 3.7 常见误区 / 风险提醒

| 误区 | 更稳妥的判断 |
|------|--------------|
| “功能最强的工具就是最佳选择” | 还要看团队迁移成本和治理能力 |
| “统一一个工具就能统一方法” | 没有共享规则和模板，统一工具也会各用各的 |
| “先买了再说，流程以后补” | 没有规则、验收和边界，工具越强风险越大 |
| “IDE 工具和 CLI 工具只能二选一” | 多数成熟团队最终都会组合使用 |

**风险提醒：**  
如果团队在没有明确工作流和权限边界前就引入高自主度工具，后面补治理的成本会更高。

## 3.8 最小练习

用下面 5 个问题给你当前团队做一次快速选型判断：

1. 团队主要需求是补全，还是复杂任务自动化？
2. 大家愿不愿意切换 IDE 或增加 CLI 工作流？
3. 项目是否需要共享 Rules、Skills、Hooks？
4. 数据和权限边界是否允许接外部工具？
5. 团队更需要“先低成本上手”还是“尽快形成重型自动化能力”？

如果能答完这 5 个问题，再去看工具列表，你会比直接比功能清楚得多。

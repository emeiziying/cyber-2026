# 第二章：Vibe Coding 实战

> **适合人群：** 工程师
> **学习目标：** 掌握 Vibe Coding 的实践方法，体验意图驱动开发的心流状态

---

## 2.1 进入 Vibe Coding 的心法

**传统编程思维 vs Vibe Coding 思维：**

```
传统思维：
"我要写一个 Express 路由，处理 POST /api/users，验证邮箱格式，
插入数据库，返回 201，错误时返回 400..."

Vibe Coding 思维：
"给我一个用户注册接口，验证邮箱，存到 PostgreSQL，
遵循我们项目的 RESTful 规范"
```

**三个核心姿势：**
1. **描述目标，不描述步骤** — 说"做什么"，不说"怎么做"
2. **快速迭代，不求一步到位** — 先跑起来，再逐步完善
3. **用语言审查，不用眼睛审查** — 告诉 AI "解释这段代码给我听"

---

## 2.2 Prompt 质量决定输出质量

### 低质量 Prompt（模糊）
```
帮我写一个登录功能
```

### 高质量 Prompt（上下文 + 约束 + 期望）
```
我的项目用 Node.js + Express + JWT，数据库是 PostgreSQL（通过 Prisma ORM）。

请实现一个用户登录接口：
- POST /api/auth/login
- 接收 { email, password }
- 用 bcrypt 验证密码
- 成功返回 JWT token（有效期 7 天），结构参考 src/utils/jwt.ts
- 失败统一返回 { error: "Invalid credentials" }，状态码 401
- 不要在错误信息中区分"邮箱不存在"和"密码错误"（安全考虑）
```

### Prompt 优化公式
```
[项目技术栈] + [具体功能] + [关键约束] + [参考现有代码] + [边界条件]
```

## 2.3 实战演练：从零到原型

以下是一个典型 Vibe Coding 会话示意，展示如何快速构建功能：

### Step 1：描述整体目标
```
我要构建一个简单的 Markdown 笔记应用后端。
技术栈：Node.js + Fastify + SQLite（better-sqlite3）
需要：笔记的 CRUD + 全文搜索
先帮我搭建项目结构
```

### Step 2：迭代细化
```
好的，现在帮我实现搜索接口，要支持标题和内容的模糊搜索，
用 SQLite FTS5 虚拟表，性能比 LIKE 查询好
```

### Step 3：发现问题，继续对话
```
搜索结果没有高亮，能在返回里加上匹配关键词的位置信息吗？
参考 SQLite snippet() 函数
```

### Step 4：Review 关键逻辑
```
解释一下 FTS5 的 rank 算法，以及为什么你选择用 BM25 而不是默认排序？
```

## 2.4 Vibe Coding 的适用场景矩阵

| 场景 | 适合度 | 说明 |
|------|--------|------|
| 快速原型 / PoC | ⭐⭐⭐⭐⭐ | 完全放开，快速出结果 |
| CRUD 接口 | ⭐⭐⭐⭐⭐ | 高度模式化，AI 表现稳定 |
| 单元测试编写 | ⭐⭐⭐⭐⭐ | 给 AI 看实现代码，让它写测试 |
| 数据处理脚本 | ⭐⭐⭐⭐ | 适合，但要 Review 边界条件 |
| 复杂业务逻辑 | ⭐⭐⭐ | 需要明确说明业务规则，严格 Review |
| 安全相关代码 | ⭐⭐ | 必须深度 Review，不能盲信 |
| 性能关键路径 | ⭐⭐ | 需要 benchmark 验证，不能只看代码 |
| 算法核心实现 | ⭐⭐ | 适合生成，不适合直接使用 |

## 2.5 常用 Vibe Coding 指令模式

```bash
# 生成代码
"实现 [功能]，参考 [现有文件]，遵循 [约束]"

# 解释代码
"解释这段代码的工作原理，特别是 [具体部分]"

# 重构代码
"重构这段代码，使其 [更清晰/更高效/符合某模式]，不要改变行为"

# 调试
"这段代码报错：[错误信息]。我的预期是 [xxx]，实际是 [xxx]，帮我找原因"

# 补充测试
"为这个函数写单元测试，覆盖正常流程、边界条件和异常情况"

# 代码 Review
"Review 这段代码，重点关注安全漏洞和边界条件处理"
```

---

## 2.6 知道什么时候"退出" Vibe 模式

当出现以下情况时，切换回精确控制模式：

- AI 第三次生成的结果仍然不对 → 停下来，自己分析问题本质
- 涉及金融、支付、权限控制 → 必须逐行 Review
- AI 的解释让你无法理解 → 不要用这段代码，重新设计
- 修改影响面很大的核心模块 → 先写测试，再改代码
